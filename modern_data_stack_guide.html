概要・アーキテクチャ

📌 このガイドで学ぶこと

モダンデータスタック（MDS）は、クラウドネイティブなデータ処理を支える技術スタックの総称です。このガイドでは以下の技術を ハンズオン形式 で学びます。

🔧 dbt (data build tool)

  • SQL によるデータ変換モデル
  • テスト機能（品質保証）
  • ドキュメント自動生成
  • Semantic Layer (MetricFlow)

🧊 Apache Iceberg

  • オープンテーブルフォーマット
  • タイムトラベル（履歴参照）
  • スキーマ進化
  • ACID トランザクション

        condition: service_healthy

データ取り込み（Iceberg Raw レイヤー）

Python スクリプトが Trino Python クライアント経由で Trino に接続し、iceberg.raw スキーマに3つの Iceberg テーブルを作成・データ投入します。

💡 なぜ Trino 経由でデータ投入するのか？

Trino の Iceberg コネクタを使うことで、MinIO への Parquet ファイル保存と Iceberg カタログへのメタデータ登録が自動的に行われます。PyIceberg を別途インストールする必要がありません。

scripts/02_ingest_data.py
python 
"""
CSV / Parquet データを Iceberg (iceberg.raw) へ取り込むスクリプト
実行: python scripts/02_ingest_data.py
"""
import pandas as pd
import pyarrow.parquet as pq
import trino

# ── Trino 接続 ──────────────────────────────────────────
conn = trino.dbapi.connect(
    host="localhost",
    port=8080,
    user="trino",
    catalog="iceberg",
)
cur = conn.cursor()


def run(sql: str) -> list:
    """SQL を実行して結果を返す"""
    cur.execute(sql)
    try:
        return cur.fetchall()
    except Exception:
        return []


def create_raw_schema():
    """raw スキーマを作成（存在しない場合）"""
    run("""
        CREATE SCHEMA IF NOT EXISTS iceberg.raw
        WITH (location = 's3://warehouse/raw')
    """)
    print("✅ iceberg.raw スキーマ準備完了")


def ingest_customers():
    """customers.csv → iceberg.raw.customers"""
    print("\n📥 customers を取り込み中...")
    run("DROP TABLE IF EXISTS iceberg.raw.customers")
    run("""
        CREATE TABLE iceberg.raw.customers (
            customer_id    INTEGER,
            customer_name  VARCHAR,
            email          VARCHAR,
            city           VARCHAR,
            segment        VARCHAR,
            created_at     VARCHAR
        ) WITH (format = 'PARQUET', location = 's3://warehouse/raw/customers')
    """)

    df = pd.read_csv("data/customers.csv")
    values = ", ".join(
        f"({r.customer_id}, '{r.customer_name}', '{r.email}', "
        f"'{r.city}', '{r.segment}', '{r.created_at}')"
        for r in df.itertuples()
    )
    run(f"INSERT INTO iceberg.raw.customers VALUES {values}")
    print(f"   → {len(df)} 件 投入完了")


def ingest_orders():
    """orders.csv → iceberg.raw.orders"""
    print("\n📥 orders を取り込み中...")
    run("DROP TABLE IF EXISTS iceberg.raw.orders")
    run("""
        CREATE TABLE iceberg.raw.orders (
            order_id      INTEGER,
            customer_id   INTEGER,
            product_id    INTEGER,
            order_date    VARCHAR,
            quantity      INTEGER,
            unit_price    DECIMAL(10, 2),
            status        VARCHAR,
            total_amount  DECIMAL(10, 2)
        ) WITH (format = 'PARQUET', location = 's3://warehouse/raw/orders')
    """)

    df = pd.read_csv("data/orders.csv")
    values = ", ".join(
        f"({r.order_id}, {r.customer_id}, {r.product_id}, "
        f"'{r.order_date}', {r.quantity}, {r.unit_price}, "
        f"'{r.status}', {r.total_amount})"
        for r in df.itertuples()
    )
    run(f"INSERT INTO iceberg.raw.orders VALUES {values}")
    print(f"   → {len(df)} 件 投入完了")


def ingest_products():
    """products.parquet → iceberg.raw.products"""
    print("\n📥 products (Parquet) を取り込み中...")
    run("DROP TABLE IF EXISTS iceberg.raw.products")
    run("""
        CREATE TABLE iceberg.raw.products (
            product_id      INTEGER,
            product_name    VARCHAR,
            category        VARCHAR,
            price           DECIMAL(10, 2),
            stock_quantity  INTEGER,
            created_at      VARCHAR
        ) WITH (format = 'PARQUET', location = 's3://warehouse/raw/products')
    """)

    df = pq.read_table("data/products.parquet").to_pandas()
    values = ", ".join(
        f"({r.product_id}, '{r.product_name}', '{r.category}', "
        f"{r.price}, {r.stock_quantity}, '{r.created_at}')"
        for r in df.itertuples()
    )
    run(f"INSERT INTO iceberg.raw.products VALUES {values}")
    print(f"   → {len(df)} 件 投入完了")


if __name__ == "__main__":
    print("🚀 データ取り込みを開始します...\n")
    create_raw_schema()
    ingest_customers()
    ingest_orders()
    ingest_products()
    print("\n🎉 取り込み完了！")
    conn.close()

profiles.yml

Trino への接続設定です。database: iceberg とすることで、すべてのテーブルが Iceberg カタログに作成されます。

dbt/profiles.yml
yaml 
modern_data_stack:
  target: dev
  outputs:
    dev:
      type: trino
      method: none        # 認証なし（開発環境）
      host: localhost
      port: 8080
      user: trino
      database: iceberg   # Iceberg カタログを使用
      schema: dwh         # デフォルトスキーマ（+schema で上書きされる）
      threads: 4

macros/generate_schema_name.sql

デフォルトでは dbt は {target_schema}_{custom_schema} のようなスキーマ名を生成します。このマクロで +schema の値をそのまま使うよう変更します。

dbt/macros/generate_schema_name.sql
sql 
-- +schema に指定した名前をそのままスキーマ名として使用するマクロ
{% macro generate_schema_name(custom_schema_name, node) -%}
    {%- if custom_schema_name is none -%}
        {{ target.schema }}
    {%- else -%}
        {{ custom_schema_name | trim }}
    {%- endif -%}
{%- endmacro %}

models/sources.yml

Python スクリプトが作成した Raw テーブルを dbt の ソース として定義します。dbt モデルから {{ source('raw', 'customers') }} で参照できるようになります。

dbt/models/sources.yml
yaml 
version: 2

sources:
  - name: raw
    description: "Python スクリプトで投入した生データ（Iceberg テーブル）"
    database: iceberg
    schema: raw
    tables:
      - name: customers
        description: "顧客マスタ（10件）"

      - name: orders
        description: "注文トランザクション（15件）"

      - name: products
        description: "商品マスタ（Parquet から生成）"

datalake 層モデル

🏞️ datalake

Raw ソースから Iceberg テーブルとしてコピーします。データの形式変換（型キャスト） を最小限行うレイヤーです。メダリオンアーキテクチャの Bronze → Silver に相当します。

mart 層モデル（分析用集計）

🏪 mart

dwh 層のデータを結合・集計し、BI ツールやレポートが直接クエリできる形に加工します。

models/mart/mart_sales_summary.sql

カテゴリ別・ステータス別の売上サマリーテーブルです。

dbt/models/mart/mart_sales_summary.sql
sql 
{{
  config(
    materialized = 'table',
    properties = {
      "format": "'PARQUET'",
      "partitioning": "ARRAY['status']"
    }
  )
}}

-- mart層: カテゴリ別・ステータス別の売上サマリー
WITH orders AS (
    SELECT * FROM {{ ref('stg_orders') }}
),

products AS (
    SELECT * FROM {{ ref('stg_products') }}
),

joined AS (
    SELECT
        o.order_date,
        date_format(CAST(o.order_date AS TIMESTAMP), '%Y-%m')  AS order_month,
        p.category,
        o.status,
        COUNT(o.order_id)         AS order_count,
        SUM(o.total_amount)       AS total_revenue,
        AVG(o.total_amount)       AS avg_order_value
    FROM orders o
    LEFT JOIN products p ON o.product_id = p.product_id
    GROUP BY 1, 2, 3, 4
)

SELECT * FROM joined

models/mart/mart_customer_ltv.sql

顧客ライフタイムバリュー（LTV）分析テーブルです。完了済み注文のみを対象に集計します。

dbt/models/mart/mart_customer_ltv.sql
sql 
-- mart層: 顧客ライフタイムバリュー分析
WITH orders AS (
    SELECT * FROM {{ ref('stg_orders') }}
),

customers AS (
    SELECT * FROM {{ ref('stg_customers') }}
),

-- 完了済み注文のみ集計
completed_orders AS (
    SELECT
        customer_id,
        COUNT(order_id)       AS total_orders,
        SUM(total_amount)     AS total_spend,
        AVG(total_amount)     AS avg_order_value,
        MIN(order_date)       AS first_order_date,
        MAX(order_date)       AS last_order_date
    FROM orders
    WHERE is_completed = TRUE
    GROUP BY customer_id
),

final AS (
    SELECT
        c.customer_id,
        c.customer_name,
        c.city,
        c.segment,
        c.is_premium,
        COALESCE(co.total_orders, 0)    AS total_orders,
        COALESCE(co.total_spend, 0.0)   AS lifetime_value,
        co.avg_order_value,
        co.first_order_date,
        co.last_order_date,
        -- アクティブ期間（日数）
        CASE
            WHEN co.first_order_date IS NOT NULL
             AND co.last_order_date  IS NOT NULL
            THEN date_diff('day', co.first_order_date, co.last_order_date)
            ELSE 0
        END AS days_active
    FROM customers c
    LEFT JOIN completed_orders co ON c.customer_id = co.customer_id
)

SELECT * FROM final

dbt テスト

dbt には2種類のテストがあります。

📋 汎用テスト（Generic Tests）

schema.yml で宣言的に設定するテストです。

  • unique — 重複なし
  • not_null — NULL なし
  • accepted_values — 許容値の確認
  • relationships — 外部キー整合性

🔍 カスタムテスト（Singular Tests）

tests/ ディレクトリに SQL ファイルを置きます。

クエリが1行以上を返すとテスト失敗という仕組みです。複雑なビジネスロジックを検証できます。

tests/assert_no_negative_amount.sql

合計金額が負値になっていないことを確認するカスタムテストです。

-- カスタムテスト: 合計金額が負の値でないことを確認

    order_id,
    total_amount
FROM {{ ref('stg_orders') }}
WHERE total_amount < 0

dbt Semantic Layer（MetricFlow）

dbt 1.6+ に統合された MetricFlow は、ビジネス指標をコードで定義し、一貫したメトリクスを提供します。「どのテーブルを JOIN するか」「どう集計するか」をセマンティックモデルとして記述することで、BI ツールや分析者が同じ定義のメトリクスを利用できます。

🔑 Semantic Layer の3つの構成要素

エンティティ (Entities)

テーブルの主キー・外部キーを定義

ディメンション (Dimensions)

グループ化に使う列（時間・カテゴリ）

メジャー (Measures)

集計する値（SUM/COUNT/AVG）

models/mart/mart_metrics.yml

dbt/models/mart/mart_metrics.yml
yaml 
semantic_models:
  - name: orders
    description: "注文データのセマンティックモデル"
    model: ref('stg_orders')
    defaults:
      agg_time_dimension: order_date

    # ── エンティティ ────────────────────────────────────
    entities:
      - name: order
        type: primary
        expr: order_id
      - name: customer
        type: foreign
        expr: customer_id

    # ── ディメンション ──────────────────────────────────
    dimensions:
      - name: order_date
        type: time
        type_params:
          time_granularity: day
      - name: status
        type: categorical
      - name: is_completed
        type: categorical

    # ── メジャー ────────────────────────────────────────
    measures:
      - name: order_count
        description: "注文件数"
        agg: count
        expr: order_id
      - name: total_revenue
        description: "売上合計"
        agg: sum
        expr: total_amount
      - name: avg_order_value
        description: "平均注文金額"
        agg: average
        expr: total_amount

# ── メトリクス定義 ────────────────────────────────────
metrics:
  - name: total_revenue
    label: "売上合計"
    description: "全注文の売上合計金額"
    type: simple
    type_params:
      measure: total_revenue

  - name: order_count
    label: "注文件数"
    description: "全注文件数"
    type: simple
    type_params:
      measure: order_count

  - name: revenue_per_order
    label: "1注文あたり売上"
    description: "1件あたりの平均売上金額（Ratio メトリクス）"
    type: ratio
    type_params:
      numerator: total_revenue
      denominator: order_count

Semantic Layer を操作する

Terminal — MetricFlow コマンド
bash 
# 定義したメトリクス一覧を表示

# 出力例:
# ┌─────────────────────┬────────────────────────┐
# │ Name                │ Label                  │
# ├─────────────────────┼────────────────────────┤

# │ order_count         │ 注文件数               │

💡 Semantic Layer のメリット

同じ「売上合計」をどのチームが計算しても、同じ SQL ロジックが使われます。BI ツール側で個別に集計ロジックを書く必要がなくなり、メトリクスの定義がコードで一元管理されます。

Apache Iceberg の特徴的な機能

Iceberg がオープンテーブルフォーマットとして Parquet/CSV などのシンプルなファイルと異なる点を実際に確認します。

🕐 タイムトラベル

過去の任意のスナップショットを参照できます

🔄 スキーマ進化

テーブルの列追加・削除が安全に行えます

📜 テーブル履歴

スナップショット一覧で変更履歴を追跡できます

① テーブル履歴・スナップショット確認

Trino CLI
sql 
-- テーブルのスナップショット履歴を確認
SELECT snapshot_id, committed_at, operation, summary
FROM iceberg.dwh."stg_orders$snapshots"
ORDER BY committed_at;

-- テーブル変更ログ（ファイルレベルの変更）
SELECT *
FROM iceberg.dwh."stg_orders$history";

② タイムトラベル（過去データを参照）

Trino CLI
sql 
-- ① 現在の snapshot_id を確認
SELECT snapshot_id, committed_at
FROM iceberg.dwh."stg_orders$snapshots"
ORDER BY committed_at;

-- ② データを更新（新しいスナップショットを作成）
--    例: dbt run --select mart を再実行してスナップショットを増やす

-- ③ 特定のスナップショット時点のデータを参照
SELECT COUNT(*) FROM iceberg.mart.mart_customer_ltv
FOR VERSION AS OF <snapshot_id>;    -- snapshot_id を①の値で置換

-- ④ 特定の日時時点のデータを参照
SELECT COUNT(*) FROM iceberg.mart.mart_customer_ltv
FOR TIMESTAMP AS OF TIMESTAMP '2024-04-30 00:00:00 UTC';

③ スキーマ進化（列の追加・削除）

Trino CLI
sql 
-- 列を追加
ALTER TABLE iceberg.dwh.stg_orders
ADD COLUMN discount DECIMAL(10, 2);

-- 列が追加されたことを確認
DESCRIBE iceberg.dwh.stg_orders;

-- 追加した列は既存データには NULL として扱われる
SELECT order_id, total_amount, discount
FROM iceberg.dwh.stg_orders
LIMIT 3;

-- 不要な列を削除
ALTER TABLE iceberg.dwh.stg_orders
DROP COLUMN discount;

④ ファイル管理情報の確認

Trino CLI
sql 
-- テーブルが保存している Parquet ファイル情報
SELECT file_path, record_count, file_size_in_bytes
FROM iceberg.mart."mart_sales_summary$files";

-- テーブルのパーティション情報（status でパーティション）
SELECT * FROM iceberg.mart."mart_sales_summary$partitions";
🧊 なぜ Iceberg なのか？

従来の Parquet/CSV ファイルはただのファイルに過ぎませんが、Iceberg は メタデータレイヤー を持ちます。これにより「ファイルを書き換えることなく列を追加」「過去のデータを参照」「複数プロセスからの同時書き込みを安全に制御」が可能になります。実データはそのまま Parquet ファイルとして MinIO に保存されています。
